# ADR 001: コアデータモデル設計

## ステータス
承認済み

## コンテキスト
Agile Metrics Hub は複数のPMツール（ClickUp, Jira, Linear, Asana）からタスクデータを取得し、統一的なメトリクスを計算する。各PMツールのデータモデルは大きく異なるため、統一モデルの設計が必要。

## 決定

### 1. Adapter パターンで PMツール差分を吸収
- `PMToolAdapter` インターフェースを定義し、各PMツール固有のAPIを抽象化
- `Registry` パターンで実行時にアダプターを登録・検索
- 新規PMツール追加時は `AdapterFactory` の実装のみ（5行以内で登録可能）

### 2. スナップショットベースのメトリクス計算
- バーンダウン/累積フローは日次スナップショット（`snapshots` テーブル）から計算
- タスク履歴APIが存在しないPMツール（ClickUp等）にも対応するため、ポーリング+Webhookでスナップショットを蓄積
- リードタイム/サイクルタイムは `task_events` テーブルから計算

### 3. APIキーの暗号化保存
- AES-256-GCM で暗号化して `BYTEA` カラムに保存
- 暗号化キーは環境変数 `ENCRYPTION_KEY` で管理
- ランダムnonce生成により同一平文でも毎回異なる暗号文を生成

### 4. sqlc によるコード生成
- ORM ではなく sqlc を採用。SQLを直接記述し、型安全なGoコードを自動生成
- 複雑なクエリ（メトリクス集計等）もSQLの表現力をフル活用可能

## 影響
- PMツール追加のコスト: インターフェース実装のみ（低コスト）
- スナップショット蓄積: ストレージが時間とともに増加 → パーティショニングで対応予定
- 暗号化オーバーヘッド: APIキー取得時に復号が必要だが、頻度は低いため無視可能

## 代替案
- **ORM (GORM/ent)**: 型安全だがSQLの表現力が制限される。メトリクス集計クエリが複雑なためsqlcを選択
- **イベントソーシング**: 完全な履歴保持が可能だが、MVP段階ではオーバーエンジニアリング
